0. 教程总览：客户端里的完整工作流

这份教程不从终端命令开始，而是从真实的 Codex 客户端使用场景开始：打开客户端、选择项目、发起需求、审阅过程、打开预览、完成交付。

01打开客户端并选择项目
02发起一条好需求
03让 Codex 先分析
04审阅命令和文件改动
05打开预览并验收
06总结交付和复盘
07沉淀到 AGENTS.md
08形成团队工作流
核心心智模型：Codex 客户端不是一个“聊天框加代码生成器”，而是一个可协作的工程工作台。你给目标和边界，Codex 负责读项目、改文件、跑命令、打开预览、解释结果；你负责把关方向、权限和验收。

1. 客户端地图篇：先知道每个区域负责什么

1.1 打开客户端后先认识这些

线程一次任务或一组相关任务的上下文。尽量一个线程解决一个明确目标。
项目/工作区Codex 当前能读取和修改的代码范围，开始前要确认目录选对。
输入框给目标、补充约束、纠偏、要求验证的地方。
执行过程读取文件、运行命令、编辑文件、打开浏览器都会留下过程。
文件改动重点审阅 Codex 改了什么，是否超出需求。
预览地址前端、文档或静态页面最终都应能打开检查。

1.2 客户端用户最常用的 8 个动作

动作什么时候用你要关注什么
新建/继续线程开始一个需求或接着做旧任务一个线程不要混太多目标。
选择项目目录让 Codex 读写某个仓库确认目录选对。
发送需求描述目标和验收标准说清楚不要改什么。
追加补充Codex 跑偏或你想到新约束及时纠偏。
审批命令安装依赖、跑服务、联网等不懂的命令先让它解释。
查看文件改动Codex 修改后看是否有无关改动。
打开预览前端或文档任务完成后桌面端和移动端都要看。
要求复盘任务结束后把重复规则沉淀到 AGENTS.md。

2. 设置篇：把 Codex 客户端调成适合自己的工作台

设置区域推荐做法为什么重要
General设置文件打开方式、输出显示偏好、多行输入习惯减少日常摩擦。
Notifications开启任务完成提醒长任务不用一直盯着。
Appearance选择舒服的主题、字体、强调色客户端会长时间使用。
Agent configuration保持默认权限起步，再按项目调整初学阶段更稳。
Git统一分支名、commit message、PR 描述偏好团队协作更一致。
Integrations & MCP只连接真正常用的工具避免权限和上下文变复杂。
Browser use允许常用本地预览地址前端项目验收更顺。
Personalization选择默认性格并写自定义指令让 Codex 按你的偏好工作。

2.1 Personalization 怎么写才有用

每次收到我的需求后，先用 2-4 句话说明你对需求的理解和影响范围，再开始实现。
完成实现后必须给出：改动摘要、验证结果、当前可预览地址、其他建议。
不要做无关重构。修改前请优先阅读项目现有写法并沿用。
客户端用户不需要一开始就学习 config.toml。初学阶段优先使用 Settings；熟悉后再了解个人配置和项目配置。

3. 需求表达篇：在客户端里发一条“能交付”的需求

3.1 第一条消息模板

请先分析再实现。

目标：
我要你实现/修复/优化什么。

上下文：
相关页面、文件、业务规则、已有问题。

边界：
不要改什么，不要重构什么，哪些行为必须保持。

验收：
需要跑哪些测试，或者我应该通过哪个预览地址检查。

交付：
最后请给我改动摘要、验证结果、当前可预览地址和其他建议。

3.2 一个更像真实客户端需求的例子

请先分析再实现。

目标：
优化价格页在移动端的卡片排版。

边界：
不要改接口和价格数据，不要重做整体视觉风格。

验收：
桌面端和移动端都要检查，重点看按钮文字是否溢出、卡片高度是否跳动。

交付：
完成后打开本地预览，给我可预览地址，并总结改动和验证结果。

3.3 中途怎么纠偏

先暂停实现，解释一下你为什么要改这个文件。
这个方向不对，请回到原需求：只修移动端排版，不要调整桌面端视觉。
在继续之前，请先列出你已经改过的文件和每个文件的目的。

4. AGENTS.md 篇：让客户端每次都记住项目规则

AGENTS.md 是给 Codex 看的项目说明书。你不用每次重复告诉它项目怎么启动、怎么测试、怎么验收，这些规则可以写进去，让 Codex 每次进入项目时自动参考。

4.1 适合写进去的内容

  • 项目结构和关键目录。
  • 本地启动、构建、测试、lint 命令。
  • 设计规范和组件复用要求。
  • 不允许改动的区域。
  • 权限、安全、数据相关注意事项。
  • 每次交付必须包含的内容。

4.2 客户端用户模板

# AGENTS.md

## 客户端工作流要求
- 收到需求后，先分析理解，再开始实现。
- 实现前优先阅读现有代码和设计风格。
- 不做无关重构，不覆盖用户已有改动。
- 涉及前端页面时，完成后必须提供当前可预览地址。

## 交付要求
- 最终回复包含：改动摘要、验证结果、当前可预览地址、其他建议。
- 如果无法运行测试或预览，说明原因和风险。

4.3 让 Codex 帮你维护

这次任务中有哪些规则值得沉淀到 AGENTS.md？请给出可直接追加的内容。

5. 权限安全篇：客户端里哪些弹窗要认真看

Sandbox限制 Codex 能访问和修改哪些文件。
ApprovalCodex 运行某些命令前需要你确认。
Network access控制是否允许访问外网、下载依赖、查资料或调用服务。
你的角色给 Codex 安全边界，让它先解释再执行高风险操作。
Codex 想做什么建议
读取项目文件通常允许。
修改工作区内需求相关文件通常允许，但要看 diff。
启动本地开发服务通常允许。
安装依赖先让它解释为什么需要。
联网搜索或下载确认是否必要。
删除文件谨慎，先问清影响。
修改密钥、生产配置、支付、权限逻辑保守处理，逐步审查。
git reset、强推、清理大量文件不要轻易允许。

6. 预览验收篇：客户端用户一定要学会看结果

6.1 什么任务必须要预览

  • 前端页面。
  • 静态 HTML 教程。
  • 图片、样式、布局相关修改。
  • 表单、弹窗、导航、支付流程等用户路径。
  • 文档或教程页面。

6.2 让 Codex 主动给预览地址

完成后请启动或打开预览，并在最终回复里给我当前可预览地址。
请不要只告诉我代码改了，请打开预览确认用户实际看到的效果。

6.3 验收重点

类型重点
页面布局是否溢出、重叠、错位、移动端不可读。
交互功能按钮、表单、弹窗、导航是否能用。
数据状态loading、empty、error、长文本是否正常。
视觉一致性是否沿用已有设计系统。
文案是否清楚、短、没有教程腔。

7. 项目实战篇：用客户端跑完一次真实交付

建议选一个真实但范围小的任务，例如修复移动端 UI、给表单加校验，或者把一份大纲改成可预览的教程页面。

第 1 步先让 Codex 阅读项目结构和相关文件，不急着改。
第 2 步确认方向后开始实现，要求保持改动范围小。
第 3 步要求打开预览，检查桌面端和移动端。
第 4 步让 Codex 自查无关改动、遗漏状态和未验证项。
第 5 步最终交付：摘要、验证结果、预览地址、其他建议。
请先阅读项目结构、启动方式和与这个需求相关的文件。先不要改代码，告诉我你的理解、涉及文件、风险点和验证方式。
方向确认。请开始实现，保持改动范围小，沿用现有组件和样式。完成后运行相关检查。
请打开预览，检查桌面端和移动端。重点看是否有文字溢出、布局错位、交互不可用。

8. 高效技巧篇：客户端用户的进阶习惯

8.1 适合交给 Codex 的任务

  • 读代码并总结模块关系。
  • 修小 bug。
  • 补测试。
  • 做第一轮 code review。
  • 根据报错定位原因。
  • 生成教程、说明文档、变更摘要。
  • 把零散规则整理成 AGENTS.md。

8.2 每次任务结束后做 3 件事

请总结这次任务：
1. 哪些做法有效？
2. 哪些问题下次要避免？
3. 哪些规则值得写进 AGENTS.md 或个人自定义指令？

9. 附录：CLI 和其他入口放到最后

这套教程面向 Codex 客户端用户，所以 CLI 不作为主线。可以在附录里简短说明：CLI 适合习惯终端的开发者，IDE Extension 适合和编辑器深度结合，Web/Cloud 适合云端并行任务或 GitHub 任务。

对本教程读者来说，第一优先级是掌握客户端里的：需求表达、权限确认、文件改动审阅、预览验收、最终交付。
